Durante o desenvolvimento do projeto, optamos por aplicar uma versão adaptada do framework Scrum com o objetivo de organizar melhor as tarefas. Essa decisão foi consenso entre a equipe,que queria ir além do escopo do jogo original proposto pelo professor.
## Papéis Definidos

- **Product Owner**: A especificação inicial foi baseada no escopo proposto pelo professor. A equipe colaborativamente sugeriu e aprovou novas funcionalidades. 
- **Scrum Master**: Thiago atuou como facilitador do processo marcando as reuniões semanais e coordenando o time, mas a equipe inteira teve a responsabilidade de engajar os colegas e ajudar no andamento do desenvolvimento.
- **Desenvolvedores**: Todos os membros da equipe participaram ativamente do processo técnico, sendo responsáveis pela implementação das funcionalidades semanais.

## Processo de Trabalho

1. **Planejamento Inicial**  
   A equipe se reuniu para ler e compreender as especificações propostas. A plataforma do Miro foi utilizada para organizar visualmente as ideias, onde foi construído um quadro branco listando as funcionalidades desejadas para o jogo.

2. **Organização das Tarefas**  
   A partir das ideias discutidas, criamos um quadro no Trello, estruturado na forma de um kanban board. Nele, as funcionalidades do jogo foram divididas em tarefas menores, associadas a marcos de desenvolvimento por semana.

3. **Sprints Semanais**  
   Cada sprint teve duração de uma semana. As tarefas foram atribuídas aos membros da equipe no início da semana e deveriam ser concluídas até sábado de cada uma delas.

4. **Code Review e Planejamento Contínuo**  
   Aos sábados, foram realizadas reuniões gerais para revisar os códigos produzidos por cada membro. O objetivo era garantir que todos compreendessem o funcionamento geral do projeto, além de discutir melhorias e ajustes de performance. Após a revisão, as tarefas da próxima semana eram planejadas com base no progresso atual, incluindo as sugestões feitas pelos demais membros.

5. **Uso de Issues para rastreamento de problemas**  
   Durante o desenvolvimento, foi adotado o sistema de Issues do GitHub como forma de registrar bugs, pendências técnicas e sugestões de melhoria. Isso permitiu uma separação clara entre tarefas de funcionalidade e correções técnicas.  
   Cada issue era simplificado para uma descrição do problema, o pedaço do código relevante e o comportamento esperado, essa funcionalidade foi útil principalmente para sugerir melhorias.
   Atribuímos etiquetas como “bug” e "enhancement" para facilitar o acompanhamento e priorização. Algumas dessas issues foram tratadas dentro das sprints, enquanto outras ficaram registradas para correção futura.

6. **Comunicação**  
   Por conta do tamanho reduzido da equipe, foi consenso geral não realizar reuniões diárias (dailies). A comunicação do progresso individual foi feita de forma assíncrona pelo grupo de WhatsApp, onde cada membro informava quando concluía sua parte, isso foi feito com o intuito de aumentar a dinamicidade dos sprints.

## Considerações Finais

O uso adaptado da metodologia Scrum contribuiu para uma melhor organização e visibilidade do progresso do projeto. A divisão em sprints e o processo de revisão coletiva foram fundamentais para garantir um código coeso e um aprendizado compartilhado entre todos os membros. A adoção das Issues como mecanismo de acompanhamento técnico trouxe maturidade ao processo, ajudando a manter o controle sobre problemas que surgiram durante o desenvolvimento e possibilitando um gerenciamento mais claro das correções.
